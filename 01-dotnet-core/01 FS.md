## C# Language Deep Dive

### 1. How does `Span<T>` improve performance, and when should you use it over `Memory<T>`?

**Answer:**  
`Span<T>` is a `ref struct` that provides a type‑safe view over contiguous memory (array, stackalloc, unmanaged memory). It avoids heap allocations and copying.  
`Memory<T>` is a heap‑allocatable wrapper that can be stored in fields or used in async methods.

**Performance benefits:**  
- Zero‑allocation slicing of strings or arrays.  
- Works directly with stack‑allocated buffers via `stackalloc`.  
- Eliminates the need for unsafe code in many scenarios.

**When to use `Span<T>`:**  
- Synchronous methods only (cannot be boxed or used as a generic type argument).  
- Short‑lived operations like parsing, formatting, or string manipulation.  

**When to use `Memory<T>`:**  
- When the buffer must be stored as a field or used in async/await.  

**Example – parsing a comma‑separated list without allocations:**
```csharp
public static int SumValues(ReadOnlySpan<char> csv)
{
    int total = 0;
    foreach (var segment in csv.Split(','))
    {
        if (int.TryParse(segment, out int value))
            total += value;
    }
    return total;
}

// Usage:
string data = "100,200,300,400";
int result = SumValues(data.AsSpan()); // No string.Split allocations
```

### 2. Explain the behaviour of `IAsyncEnumerable<T>` and how it differs from `Task<IEnumerable<T>>`.

**Answer:**  
`IAsyncEnumerable<T>` (C# 8+) enables streaming of data asynchronously, yielding items one at a time as they become available.  
`Task<IEnumerable<T>>` returns the entire collection only after all items are ready.

**Key differences:**

| Feature                  | `IAsyncEnumerable<T>`                    | `Task<IEnumerable<T>>`                 |
|--------------------------|------------------------------------------|----------------------------------------|
| Memory usage             | Low – streams items                      | High – must buffer entire collection   |
| First item latency       | Lower – first item arrives earlier       | Higher – wait for all items            |
| Cancellation support     | Built‑in with `CancellationToken`        | Usually not cancellable during fetch   |
| Consumer syntax          | `await foreach`                          | `foreach` after `await`                |

**Example – reading large database result set:**
```csharp
public async IAsyncEnumerable<Customer> GetCustomersAsync(
    [EnumeratorCancellation] CancellationToken cancellationToken)
{
    using var connection = new SqlConnection(connectionString);
    await connection.OpenAsync(cancellationToken);
    using var command = new SqlCommand("SELECT * FROM Customers", connection);
    using var reader = await command.ExecuteReaderAsync(cancellationToken);
    
    while (await reader.ReadAsync(cancellationToken))
    {
        yield return new Customer
        {
            Id = reader.GetInt32(0),
            Name = reader.GetString(1)
        };
    }
}

// Usage:
await foreach (var customer in GetCustomersAsync(token))
{
    Console.WriteLine(customer.Name);
}
```

### 3. What are `record` types in C# 9+ and how do they differ from classes and structs?

**Answer:**  
Records are reference types (or value types with `record struct`) that provide **value‑based equality** and built‑in immutability support.

**Characteristics:**
- **Value equality:** Two record instances with the same property values are considered equal.
- **Immutability:** Use positional records or `init`‑only setters.
- **`with` expressions:** Create a copy with modifications.
- **Compiler‑generated members:** `Equals`, `GetHashCode`, `ToString`, `Deconstruct`.

**Comparison with classes/structs:**

| Feature              | `class`            | `struct`          | `record`                 | `record struct`          |
|----------------------|--------------------|-------------------|--------------------------|--------------------------|
| Reference/Value      | Reference          | Value             | Reference                | Value                    |
| Equality             | Reference equality | Value equality (boxing risk) | Value equality      | Value equality           |
| Mutability           | Mutable by default | Mutable by default| Immutable by position    | Immutable by position    |
| Inheritance          | Yes                | No                | Yes                      | No                       |

**Example:**
```csharp
public record Person(string FirstName, string LastName);

var person1 = new Person("John", "Doe");
var person2 = new Person("John", "Doe");

Console.WriteLine(person1 == person2); // True

var updated = person1 with { LastName = "Smith" };
Console.WriteLine(updated); // Person { FirstName = John, LastName = Smith }
```

**Performance note:** Records are reference types and subject to GC. For high‑frequency scenarios, prefer `record struct` (C# 10+).

### 4. How do you implement and use a custom `awaitable` type?

**Answer:**  
Any type that exposes a `GetAwaiter()` method returning an object with `IsCompleted`, `OnCompleted(Action)`, and `GetResult()` can be awaited.

**Example – a simple delay awaiter:**
```csharp
public class Delay
{
    private readonly TimeSpan _duration;
    public Delay(TimeSpan duration) => _duration = duration;

    public TaskAwaiter GetAwaiter() => Task.Delay(_duration).GetAwaiter();
}

// Usage:
await new Delay(TimeSpan.FromSeconds(2));
Console.WriteLine("2 seconds passed.");
```

**Advanced – custom awaitable with cancellation:**
```csharp
public struct CancellableDelay
{
    private readonly TimeSpan _duration;
    private readonly CancellationToken _token;
    public CancellableDelay(TimeSpan duration, CancellationToken token)
    {
        _duration = duration;
        _token = token;
    }

    public TaskAwaiter GetAwaiter() => Task.Delay(_duration, _token).GetAwaiter();
}
```

**When useful:**  
- Creating fluent APIs for asynchronous workflows.  
- Wrapping legacy asynchronous patterns (APM/EAP) with modern `await` syntax.

### 5. Explain `ref struct` and `readonly ref struct` with an example.

**Answer:**  
A `ref struct` is allocated on the stack only and cannot be boxed or stored on the heap. This ensures it never escapes the current method’s stack frame. `readonly ref struct` adds immutability.

**Restrictions:**
- Cannot be a field of a class or non‑`ref struct`.
- Cannot be used in async methods, iterators, or lambda captures.
- Cannot implement interfaces.

**Use case – high‑performance, zero‑allocation buffer:**
```csharp
public ref struct StackBuffer
{
    private Span<byte> _buffer;
    private int _position;

    public StackBuffer(Span<byte> buffer) => _buffer = buffer;

    public void Write(ReadOnlySpan<byte> data)
    {
        data.CopyTo(_buffer[_position..]);
        _position += data.Length;
    }

    public ReadOnlySpan<byte> Written => _buffer[.._position];
}

// Usage (on the stack):
Span<byte> stackMemory = stackalloc byte[256];
var buffer = new StackBuffer(stackMemory);
buffer.Write("Hello"u8);
buffer.Write(" World"u8);
Console.WriteLine(Encoding.UTF8.GetString(buffer.Written));
```

## .NET Core / ASP.NET Core Fundamentals

### 6. Describe the request processing pipeline in ASP.NET Core and how middleware order affects behaviour.

**Answer:**  
ASP.NET Core processes HTTP requests through a **pipeline of middleware components**. Each middleware can:
- Perform pre‑processing (e.g., authentication, logging).
- Call the next middleware via `await _next(context)`.
- Perform post‑processing after the next middleware returns.

**Order matters – typical pipeline:**
```
UseExceptionHandler → UseHsts → UseHttpsRedirection → UseStaticFiles → UseRouting → UseCors → UseAuthentication → UseAuthorization → UseEndpoints
```

**Example – custom middleware that measures request duration:**
```csharp
public class TimingMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger _logger;

    public TimingMiddleware(RequestDelegate next, ILogger<TimingMiddleware> logger)
    {
        _next = next;
        _logger = logger;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        var sw = Stopwatch.StartNew();
        await _next(context);
        sw.Stop();
        _logger.LogInformation("Request {Path} took {Elapsed}ms",
            context.Request.Path, sw.ElapsedMilliseconds);
    }
}

// Registration:
app.UseMiddleware<TimingMiddleware>();
```

**Impact of wrong order:**  
Placing `UseAuthentication` after `UseAuthorization` would cause authorization to run before the user is authenticated, resulting in 401/403 errors for valid requests.

### 7. How does the Generic Host differ from the Web Host, and why was it introduced?

**Answer:**  
The **Generic Host** (`IHost`) was introduced in .NET Core 2.1 to support non‑HTTP workloads (background services, console apps, workers). The **Web Host** (`IWebHost`) is specific to HTTP‑based web applications.

| Feature                | Generic Host (`Host.CreateDefaultBuilder`) | Web Host (`WebHost.CreateDefaultBuilder`) |
|------------------------|--------------------------------------------|-------------------------------------------|
| Workload support       | Any (HTTP, gRPC, workers, message queues)  | HTTP only                                 |
| Configuration          | Unified `appsettings.json` + environment   | Web‑specific                              |
| Dependency Injection   | Same DI container                          | Same                                      |
| Logging                | Unified                                    | Unified                                   |
| Lifetime management    | `IHostedService`, `IHostApplicationLifetime` | `IApplicationLifetime` (obsolete)        |

**Why move to Generic Host?**  
- ASP.NET Core 3.0+ uses Generic Host under the hood (`ConfigureWebHostDefaults`).  
- Enables building microservices that mix HTTP and background processing.  
- Consistent configuration and startup across all .NET Core application types.

**Example – Worker Service template:**
```csharp
public class Program
{
    public static void Main(string[] args) =>
        CreateHostBuilder(args).Build().Run();

    public static IHostBuilder CreateHostBuilder(string[] args) =>
        Host.CreateDefaultBuilder(args)
            .ConfigureServices((hostContext, services) =>
            {
                services.AddHostedService<Worker>();
            });
}
```

### 8. Explain `IHostedService` and `BackgroundService` with a practical example.

**Answer:**  
`IHostedService` defines a long‑running background task. `BackgroundService` is a base class that simplifies implementation by providing an abstract `ExecuteAsync` method.

**Example – a hosted service that polls a database for pending jobs:**
```csharp
public class JobProcessor : BackgroundService
{
    private readonly IServiceProvider _services;
    private readonly ILogger<JobProcessor> _logger;

    public JobProcessor(IServiceProvider services, ILogger<JobProcessor> logger)
    {
        _services = services;
        _logger = logger;
    }

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            using var scope = _services.CreateScope();
            var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
            
            var pendingJobs = await db.Jobs
                .Where(j => j.Status == JobStatus.Pending)
                .Take(10)
                .ToListAsync(stoppingToken);

            foreach (var job in pendingJobs)
            {
                // Process job
                job.Status = JobStatus.Processed;
            }
            
            await db.SaveChangesAsync(stoppingToken);
            await Task.Delay(TimeSpan.FromSeconds(30), stoppingToken);
        }
    }
}
```

**Graceful shutdown:**  
The `stoppingToken` is triggered when the host shuts down. Ensure your code respects it to avoid abrupt termination.

### 9. What is the `Options` pattern and how does it support configuration validation?

**Answer:**  
The Options pattern provides strongly‑typed access to configuration sections. It supports validation, post‑configuration, and named options.

**Implementation steps:**

1. **Define a class matching the configuration section:**
```csharp
public class EmailSettings
{
    public string SmtpServer { get; set; } = string.Empty;
    public int Port { get; set; }
    public bool EnableSsl { get; set; }
}
```

2. **Register with validation:**
```csharp
services.AddOptions<EmailSettings>()
    .Bind(Configuration.GetSection("Email"))
    .ValidateDataAnnotations()
    .ValidateOnStart(); // Validate at app startup

// Or custom validation:
services.AddOptions<EmailSettings>()
    .Bind(Configuration.GetSection("Email"))
    .Validate(settings =>
    {
        return settings.Port is > 0 and < 65536;
    }, "Port must be between 1 and 65535");
```

3. **Inject into services:**
```csharp
public class EmailService
{
    private readonly EmailSettings _settings;
    public EmailService(IOptions<EmailSettings> options) => _settings = options.Value;
}
```

**Validation attributes on the class:**
```csharp
public class EmailSettings
{
    [Required]
    public string SmtpServer { get; set; }
    [Range(1, 65535)]
    public int Port { get; set; }
}
```

**Snapshot / Monitor support:** Use `IOptionsSnapshot<T>` for scoped changes (e.g., during a request) or `IOptionsMonitor<T>` for real‑time updates.

### 10. How do you implement health checks in ASP.NET Core for a production system?

**Answer:**  
Health checks monitor the liveness and readiness of an application and its dependencies.

**Built‑in checks and custom ones:**
```csharp
services.AddHealthChecks()
    .AddDbContextCheck<AppDbContext>("Database")
    .AddUrlGroup(new Uri("https://external-api.com/health"), "External API")
    .AddRedis("redis-connection")
    .AddCheck<CustomHealthCheck>("Custom");

// Custom health check:
public class CustomHealthCheck : IHealthCheck
{
    public Task<HealthCheckResult> CheckHealthAsync(HealthCheckContext context,
        CancellationToken cancellationToken = default)
    {
        bool isHealthy = true; // Custom logic
        return Task.FromResult(isHealthy
            ? HealthCheckResult.Healthy("All good")
            : HealthCheckResult.Unhealthy("Something wrong"));
    }
}
```

**Expose endpoints:**
```csharp
app.UseHealthChecks("/health/ready", new HealthCheckOptions
{
    Predicate = _ => true // All checks
});
app.UseHealthChecks("/health/live", new HealthCheckOptions
{
    Predicate = check => check.Tags.Contains("live")
});
```

**Integration with Kubernetes:**  
- `/health/live` – Liveness probe (restart container if failing).  
- `/health/ready` – Readiness probe (stop sending traffic if failing).

## ASP.NET Core Web API Advanced Patterns

### 11. Explain the differences between Minimal APIs and Controller‑based APIs. When would you choose each?

**Answer:**  

| Feature                | Minimal APIs (`.MapGet`, etc.)                      | Controller‑based APIs                                 |
|------------------------|-----------------------------------------------------|-------------------------------------------------------|
| Code verbosity         | Low – inline endpoint definitions                    | Higher – separate controller classes                  |
| Performance            | Slightly faster (less overhead)                      | Good but slightly more overhead                       |
| Features supported     | Filters, auth, DI, model binding (with some limits)  | Full MVC pipeline (filters, validation, formatters)   |
| OpenAPI/Swagger        | Supported with `WithOpenApi()`                       | Supported via `Swashbuckle` or `NSwag`                |
| Testability            | Harder to unit test endpoint logic directly          | Controllers are easily unit testable                  |
| Best for               | Microservices, small APIs, lambdas                   | Large RESTful APIs with complex requirements          |

**Minimal API Example:**
```csharp
app.MapGet("/products/{id}", async (int id, AppDbContext db) =>
{
    var product = await db.Products.FindAsync(id);
    return product is null ? Results.NotFound() : Results.Ok(product);
})
.Produces<Product>(StatusCodes.Status200OK)
.Produces(StatusCodes.Status404NotFound)
.WithName("GetProductById");
```

**When to use Minimal APIs:**
- Simple CRUD microservices.  
- Azure Functions or AWS Lambda style endpoints.  
- When you want to reduce ceremony.

**When to stick with Controllers:**
- Need advanced model binding (e.g., multiple file uploads).  
- Extensive use of action filters.  
- Team familiarity with MVC pattern.

### 12. How do you implement API versioning in ASP.NET Core?

**Answer:**  
Use the `Microsoft.AspNetCore.Mvc.Versioning` package (or `Asp.Versioning.Mvc` for newer versions). Versioning can be done via URL path, query string, header, or media type.

**Setup:**
```csharp
services.AddApiVersioning(options =>
{
    options.DefaultApiVersion = new ApiVersion(1, 0);
    options.AssumeDefaultVersionWhenUnspecified = true;
    options.ReportApiVersions = true;
    options.ApiVersionReader = ApiVersionReader.Combine(
        new UrlSegmentApiVersionReader(),
        new HeaderApiVersionReader("x-api-version"),
        new QueryStringApiVersionReader("api-version"));
});
```

**Controller example (URL segment):**
```csharp
[ApiController]
[Route("api/v{version:apiVersion}/[controller]")]
[ApiVersion("1.0")]
[ApiVersion("2.0")]
public class ProductsController : ControllerBase
{
    [HttpGet("{id}")]
    public IActionResult GetV1(int id) => Ok($"v1 product {id}");

    [HttpGet("{id}"), MapToApiVersion("2.0")]
    public IActionResult GetV2(int id) => Ok($"v2 product {id} with extra fields");
}
```

**Deprecation policy:**
```csharp
[ApiVersion("1.0", Deprecated = true)]
```

**Sunset policy (RFC 8594):** Add `api-supported-versions` and `api-deprecated-versions` headers to inform clients.

### 13. Describe the `ProblemDetails` standard and how to customise error responses.

**Answer:**  
`ProblemDetails` (RFC 7807) is a standard format for HTTP API error responses. ASP.NET Core automatically returns `ProblemDetails` for client errors (400–499) when `UseStatusCodePages` or `UseExceptionHandler` is enabled.

**Enabling:**
```csharp
app.UseExceptionHandler(exceptionHandlerApp =>
{
    exceptionHandlerApp.Run(async context =>
    {
        context.Response.StatusCode = StatusCodes.Status500InternalServerError;
        context.Response.ContentType = "application/problem+json";

        var pd = new ProblemDetails
        {
            Status = context.Response.StatusCode,
            Title = "An unexpected error occurred",
            Type = "https://tools.ietf.org/html/rfc7231#section-6.6.1",
            Instance = context.Request.Path
        };
        await context.Response.WriteAsJsonAsync(pd);
    });
});

app.UseStatusCodePages(async context =>
{
    if (context.HttpContext.Response.StatusCode == StatusCodes.Status404NotFound)
    {
        var pd = new ProblemDetails
        {
            Status = 404,
            Title = "Resource not found",
            Type = "https://tools.ietf.org/html/rfc7231#section-6.5.4"
        };
        await context.HttpContext.Response.WriteAsJsonAsync(pd);
    }
});
```

**Custom ProblemDetails for validation errors (via `InvalidModelStateResponseFactory`):**
```csharp
services.AddControllers()
    .ConfigureApiBehaviorOptions(options =>
    {
        options.InvalidModelStateResponseFactory = context =>
        {
            var problemDetails = new ValidationProblemDetails(context.ModelState)
            {
                Status = StatusCodes.Status400BadRequest,
                Title = "One or more validation errors occurred.",
                Instance = context.HttpContext.Request.Path
            };
            return new BadRequestObjectResult(problemDetails);
        };
    });
```

### 14. How do you secure a Web API using JWT Bearer tokens?

**Answer:**  
JWT Bearer authentication validates tokens issued by an identity provider (e.g., Auth0, Azure AD, IdentityServer).

**Configuration:**
```csharp
services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.Authority = "https://your-identity-server";
        options.Audience = "api1";
        options.TokenValidationParameters = new TokenValidationParameters
        {
            ValidateIssuer = true,
            ValidateAudience = true,
            ValidateLifetime = true,
            ValidateIssuerSigningKey = true,
            ClockSkew = TimeSpan.Zero // Eliminate default 5 min skew
        };
    });
```

**Protecting endpoints:**
```csharp
[Authorize]
[ApiController]
[Route("api/[controller]")]
public class OrdersController : ControllerBase
{
    [HttpGet]
    [Authorize(Roles = "Admin,Manager")]
    public IActionResult Get() => Ok("Secure data");
}
```

**Role/Permission mapping:**
```csharp
options.TokenValidationParameters.RoleClaimType = "role";
// Or use custom policy:
services.AddAuthorization(options =>
{
    options.AddPolicy("RequireAdmin", policy => policy.RequireClaim("permissions", "admin"));
});
```

**Important:** For SPAs, use **Authorization Code Flow + PKCE**, not implicit flow. Store tokens securely (preferably in HTTP‑only cookies via Backend‑For‑Frontend).

### 15. Explain the use of `IAsyncActionFilter`, `IAsyncResultFilter`, and `IAsyncExceptionFilter`.

**Answer:**  
Filters allow cross‑cutting concerns to be applied to action methods.

| Filter Type                | Executes                               | Use Cases                                        |
|----------------------------|----------------------------------------|--------------------------------------------------|
| `IAsyncAuthorizationFilter` | Before model binding                   | Custom auth logic                                |
| `IAsyncResourceFilter`      | After auth, before model binding       | Caching, short‑circuiting pipeline               |
| `IAsyncActionFilter`        | Before and after action execution      | Logging, validation, modifying arguments         |
| `IAsyncExceptionFilter`     | When an unhandled exception occurs     | Global exception handling, logging                |
| `IAsyncResultFilter`        | Before and after result execution      | Modify response, add headers, format result       |

**Example – action filter that measures execution time:**
```csharp
public class ExecutionTimeFilter : IAsyncActionFilter
{
    private readonly ILogger<ExecutionTimeFilter> _logger;
    public ExecutionTimeFilter(ILogger<ExecutionTimeFilter> logger) => _logger = logger;

    public async Task OnActionExecutionAsync(ActionExecutingContext context,
        ActionExecutionDelegate next)
    {
        var sw = Stopwatch.StartNew();
        var resultContext = await next();
        sw.Stop();
        
        _logger.LogInformation("Action {Action} took {Elapsed}ms",
            context.ActionDescriptor.DisplayName, sw.ElapsedMilliseconds);
    }
}

// Register globally:
services.AddControllers(options =>
{
    options.Filters.Add<ExecutionTimeFilter>();
});
```

**Example – exception filter that returns custom problem details:**
```csharp
public class ApiExceptionFilter : IAsyncExceptionFilter
{
    public Task OnExceptionAsync(ExceptionContext context)
    {
        var problem = new ProblemDetails
        {
            Status = 500,
            Title = "An error occurred while processing your request.",
            Detail = context.Exception.Message
        };
        context.Result = new ObjectResult(problem) { StatusCode = 500 };
        context.ExceptionHandled = true;
        return Task.CompletedTask;
    }
}
```

## Dependency Injection and Inversion of Control

### 16. Explain the three service lifetimes in ASP.NET Core DI and provide examples of when to use each.

**Answer:**  

| Lifetime      | Description                                                                 | Use Cases                                                                 |
|---------------|-----------------------------------------------------------------------------|----------------------------------------------------------------------------|
| **Transient** | Created each time they are requested.                                       | Lightweight, stateless services (e.g., email sender, password hasher).      |
| **Scoped**    | Created once per scope (in ASP.NET Core, per HTTP request).                  | Services that need to share state within a request (e.g., DbContext).       |
| **Singleton** | Created once and shared across the entire application lifetime.              | Stateless services with shared data, caching, configuration wrappers.       |

**Registration examples:**
```csharp
services.AddTransient<IEmailSender, SmtpEmailSender>();
services.AddScoped<IOrderService, OrderService>();
services.AddSingleton<ICacheService, RedisCacheService>();
```

**Pitfalls:**
- **Captive Dependency:** Injecting a Scoped service into a Singleton causes the scoped instance to live forever (or throws at runtime if you enable `ValidateScopes`).
- **Memory Leak:** Storing request‑specific data in a Singleton.

**Example – detecting captive dependencies in development:**
```csharp
public static IHostBuilder CreateHostBuilder(string[] args) =>
    Host.CreateDefaultBuilder(args)
        .UseDefaultServiceProvider((context, options) =>
        {
            options.ValidateScopes = context.HostingEnvironment.IsDevelopment();
            options.ValidateOnBuild = true;
        });
```

### 17. How do you register multiple implementations of the same interface and resolve them selectively?

**Answer:**  
Use `IEnumerable<T>` injection to get all implementations, or use a factory pattern / named registrations.

**Approach 1: Inject `IEnumerable<T>` and choose based on condition:**
```csharp
services.AddScoped<IPaymentProcessor, CreditCardProcessor>();
services.AddScoped<IPaymentProcessor, PayPalProcessor>();
services.AddScoped<IPaymentProcessor, BankTransferProcessor>();

public class CheckoutService
{
    private readonly IEnumerable<IPaymentProcessor> _processors;
    public CheckoutService(IEnumerable<IPaymentProcessor> processors) => _processors = processors;

    public void Process(string paymentMethod)
    {
        var processor = _processors.FirstOrDefault(p => p.CanHandle(paymentMethod));
        processor?.Process();
    }
}
```

**Approach 2: Use a factory delegate:**
```csharp
services.AddScoped<CreditCardProcessor>();
services.AddScoped<PayPalProcessor>();
services.AddScoped<IPaymentProcessor>(sp => 
{
    var context = sp.GetRequiredService<IHttpContextAccessor>();
    var method = context.HttpContext.Request.Query["method"];
    return method switch
    {
        "paypal" => sp.GetRequiredService<PayPalProcessor>(),
        _ => sp.GetRequiredService<CreditCardProcessor>()
    };
});
```

**Approach 3: Use keyed services (C# 11 / .NET 8 preview):**
```csharp
services.AddKeyedScoped<IPaymentProcessor, CreditCardProcessor>("creditcard");
services.AddKeyedScoped<IPaymentProcessor, PayPalProcessor>("paypal");

// Resolve via [FromKeyedServices] attribute in constructor
public class CheckoutService([FromKeyedServices("paypal")] IPaymentProcessor processor)
```

### 18. What is `IServiceScopeFactory` and when is it necessary?

**Answer:**  
`IServiceScopeFactory` creates a new DI scope manually. It is essential when you need to resolve scoped services outside the normal request scope (e.g., in a singleton service or background thread).

**Common scenarios:**
- Background tasks (`IHostedService`, `BackgroundService`)
- Fire‑and‑forget operations
- Parallel processing of multiple items

**Example – background service consuming scoped DbContext:**
```csharp
public class OrderProcessingService : BackgroundService
{
    private readonly IServiceScopeFactory _scopeFactory;

    public OrderProcessingService(IServiceScopeFactory scopeFactory) => _scopeFactory = scopeFactory;

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            using var scope = _scopeFactory.CreateScope();
            var dbContext = scope.ServiceProvider.GetRequiredService<AppDbContext>();
            var orders = await dbContext.Orders.Where(o => !o.Processed).ToListAsync();
            // Process...
            await Task.Delay(5000, stoppingToken);
        }
    }
}
```

**Anti‑pattern:** Do not store a scoped service reference in a singleton field. Always create a fresh scope.

### 19. How do you implement decorator pattern with .NET Core DI?

**Answer:**

### 20. Explain the difference between `IQueryable` and `IEnumerable` and why it matters for EF Core performance.

**Answer:**  
`IQueryable<T>` builds an expression tree that is translated to SQL and executed on the database server.  
`IEnumerable<T>` operates in‑memory on already materialized objects.

**Performance impact:**
```csharp
// IQueryable: filters at database, returns only matching rows
var expensiveProducts = dbContext.Products
    .Where(p => p.Price > 1000)
    .ToList();

// IEnumerable: fetches ALL products, filters in memory (bad)
IEnumerable<Product> allProducts = dbContext.Products;
var expensive = allProducts.Where(p => p.Price > 1000).ToList();
```

**Common mistake – premature materialization:**
```csharp
var query = dbContext.Products.AsEnumerable() // switches to client eval
    .Where(p => SomeComplexCSharpMethod(p));
```

**Best practice:** Keep the query as `IQueryable` until the final execution (e.g., `ToListAsync`, `FirstOrDefaultAsync`). Use projection (`Select`) to reduce data transfer.

### 21. How do you handle concurrency conflicts in EF Core?

**Answer:**  
Use **optimistic concurrency control** with a `RowVersion` column (or `ConcurrencyCheck` attribute).

**Model:**
```csharp
public class Product
{
    public int Id { get; set; }
    public string Name { get; set; }
    public decimal Price { get; set; }
    [Timestamp]
    public byte[] RowVersion { get; set; }
}
```

**During update, EF Core includes `RowVersion` in the `WHERE` clause:**
```sql
UPDATE Products SET Name = @p0, Price = @p1
WHERE Id = @p2 AND RowVersion = @p3;
```

**Handling `DbUpdateConcurrencyException`:**
```csharp
try
{
    await dbContext.SaveChangesAsync();
}
catch (DbUpdateConcurrencyException ex)
{
    foreach (var entry in ex.Entries)
    {
        var databaseValues = await entry.GetDatabaseValuesAsync();
        if (databaseValues == null)
        {
            // Entity was deleted
            return;
        }
        // Resolve conflict (e.g., client wins)
        entry.OriginalValues.SetValues(databaseValues);
        // Retry
        await dbContext.SaveChangesAsync();
    }
}
```

**Alternative – use `ConcurrencyCheck` on specific property:**
```csharp
[ConcurrencyCheck]
public decimal Price { get; set; }
```

### 24. How do you implement the **Outbox Pattern** with EF Core to guarantee message delivery?

**Answer:**  
The Outbox pattern ensures that database changes and event publication occur atomically within the same transaction.

**Steps:**
1. In the same DbContext transaction, save domain events to an `OutboxMessage` table.
2. A separate background worker reads the outbox table and publishes messages to a broker (RabbitMQ, Azure Service Bus).
3. After successful publish, mark the message as processed.

**Implementation:**
```csharp
public class OrderService
{
    private readonly AppDbContext _dbContext;
    public async Task PlaceOrder(Order order)
    {
        await using var transaction = await _dbContext.Database.BeginTransactionAsync();
        
        _dbContext.Orders.Add(order);
        
        var outboxMessage = new OutboxMessage
        {
            Id = Guid.NewGuid(),
            EventType = "OrderPlaced",
            EventData = JsonSerializer.Serialize(new OrderPlacedEvent(order.Id)),
            OccurredOn = DateTime.UtcNow
        };
        _dbContext.OutboxMessages.Add(outboxMessage);
        
        await _dbContext.SaveChangesAsync();
        await transaction.CommitAsync();
    }
}
```

**Background publisher:**
```csharp
public class OutboxPublisher : BackgroundService
{
    private readonly IServiceScopeFactory _scopeFactory;
    private readonly IMessageBus _messageBus;

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            using var scope = _scopeFactory.CreateScope();
            var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
            
            var messages = await db.OutboxMessages
                .Where(m => m.ProcessedOn == null)
                .Take(100)
                .ToListAsync(stoppingToken);
            
            foreach (var msg in messages)
            {
                await _messageBus.PublishAsync(msg.EventType, msg.EventData);
                msg.ProcessedOn = DateTime.UtcNow;
            }
            
            await db.SaveChangesAsync(stoppingToken);
            await Task.Delay(2000, stoppingToken);
        }
    }
}
```

## SQL Server Performance and Design

### 25. What are the different types of indexes in SQL Server, and when would you use each?

**Answer:**  

| Index Type               | Description                                                              | Use Cases                                                                 |
|--------------------------|--------------------------------------------------------------------------|----------------------------------------------------------------------------|
| **Clustered**            | Physically orders data rows by the index key. One per table.              | Primary key (by default), range scans, queries that return ordered data.   |
| **Non‑clustered**        | Separate structure with pointers to data rows. Up to 999 per table.       | Equality searches, covering queries, foreign key columns.                  |
| **Filtered**             | Non‑clustered index with a `WHERE` clause.                                | Columns with many NULLs, subset of data (e.g., active records only).       |
| **Columnstore**          | Stores data column‑wise, highly compressed.                               | Large data warehouse queries, aggregations over millions of rows.          |
| **Full‑Text**            | Enables linguistic searches on text columns.                              | Searching documents, product descriptions.                                 |
| **Spatial**              | Optimized for geometry/geography data.                                    | Location‑based queries, proximity searches.                                |
| **XML**                  | Indexes on XML columns.                                                   | Querying XML data efficiently.                                             |

**Index design guidelines:**
- Keep clustered index narrow and ever‑increasing (e.g., `IDENTITY`).
- Use **included columns** (`INCLUDE`) to create covering indexes and eliminate key lookups.
- Avoid over‑indexing – each index adds overhead on `INSERT/UPDATE/DELETE`.

**Example – covering index:**
```sql
CREATE NONCLUSTERED INDEX IX_Orders_CustomerId_OrderDate 
ON Orders (CustomerId) 
INCLUDE (OrderDate, TotalAmount);
```

### 26. How do you analyse and optimise a slow‑running query in SQL Server?

**Answer:**  

**Step 1: Identify expensive queries**  
- Use **Query Store** (enabled by default in recent versions) to find regressed queries.
- Use **DMVs**:
```sql
SELECT TOP 10 
    qs.total_worker_time / qs.execution_count AS avg_cpu_time,
    qs.execution_count,
    qs.total_logical_reads / qs.execution_count AS avg_logical_reads,
    SUBSTRING(st.text, (qs.statement_start_offset/2) + 1,
        ((CASE qs.statement_end_offset WHEN -1 THEN DATALENGTH(st.text) 
        ELSE qs.statement_end_offset END - qs.statement_start_offset)/2) + 1) AS query_text
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) st
ORDER BY avg_cpu_time DESC;
```

**Step 2: Examine actual execution plan**  
- Look for **Table Scans** or **Index Scans** (should be seeks for large tables).
- Check for **Key Lookups** (suggests missing included columns).
- Watch for **thick arrows** (large data movement between operators).

**Step 3: Common fixes**  
- Add missing indexes (the plan often suggests them).
- Update statistics (`UPDATE STATISTICS`).
- Rewrite query to avoid functions on indexed columns (e.g., `WHERE YEAR(OrderDate) = 2023` prevents index usage).
- Use `OPTION (RECOMPILE)` for parameter sniffing issues.

**Step 4: Use indexing hints cautiously:**
```sql
SELECT * FROM Orders WITH (INDEX(IX_Orders_CustomerId)) WHERE CustomerId = 123;
```

### 27. Explain transaction isolation levels and their impact on concurrency.

**Answer:**  

| Isolation Level        | Dirty Read | Non‑repeatable Read | Phantom Read | Concurrency Control          |
|------------------------|------------|---------------------|--------------|------------------------------|
| **Read Uncommitted**   | Yes        | Yes                 | Yes          | None (NOLOCK)                |
| **Read Committed**     | No         | Yes                 | Yes          | Lock committed rows (default)|
| **Repeatable Read**    | No         | No                  | Yes          | Locks read rows until commit |
| **Serializable**       | No         | No                  | No           | Range locks, strictest       |
| **Snapshot**           | No         | No                  | No           | Row versioning (no locks)    |

**In .NET with EF Core:**
```csharp
using var transaction = await dbContext.Database.BeginTransactionAsync(IsolationLevel.Snapshot);
// Operations...
await transaction.CommitAsync();
```

**Snapshot isolation** requires enabling at database level:
```sql
ALTER DATABASE MyDb SET ALLOW_SNAPSHOT_ISOLATION ON;
ALTER DATABASE MyDb SET READ_COMMITTED_SNAPSHOT ON; -- Changes default Read Committed behaviour
```

**Best practice:** Use `READ COMMITTED SNAPSHOT` to reduce blocking in high‑concurrency OLTP workloads.

### 29. What are the best practices for connection pooling and preventing leaks?

**Answer:**  
Connection pooling reuses a set of active database connections, dramatically improving performance.

**Best practices:**
- **Always dispose connections:** Use `using` statements or `await using`.
```csharp
await using var connection = new SqlConnection(connectionString);
await connection.OpenAsync();
```
- **Use the same connection string** for all connections to share the same pool.
- **Set appropriate pool size limits** in connection string:
```
"Server=...;Pooling=true;Min Pool Size=5;Max Pool Size=100;"
```
- **Avoid long‑running connections** – they block other requests and exhaust pool.
- **Clear pools during app restarts or failover:**
```csharp
SqlConnection.ClearPool(connection);
SqlConnection.ClearAllPools();
```

**Detecting leaks:** Monitor `NumberOfPooledConnections` and `NumberOfActiveConnections` performance counters.

**Anti‑patterns:**
- Keeping a connection open for the entire request duration.
- Storing connection objects in static fields.

### 33. How do you implement the **Circuit Breaker** pattern in .NET microservices?

**Answer:**  
The Circuit Breaker prevents cascading failures when a downstream service is unavailable. Use **Polly** library with `HttpClientFactory`.

**Configuration:**
```csharp
services.AddHttpClient("orders", client =>
{
    client.BaseAddress = new Uri("https://orders-api");
})
.AddPolicyHandler(GetCircuitBreakerPolicy());

static IAsyncPolicy<HttpResponseMessage> GetCircuitBreakerPolicy()
{
    return HttpPolicyExtensions
        .HandleTransientHttpError()
        .CircuitBreakerAsync(
            handledEventsAllowedBeforeBreaking: 3,
            durationOfBreak: TimeSpan.FromSeconds(30),
            onBreak: (outcome, timespan) => { /* log */ },
            onReset: () => { /* log */ },
            onHalfOpen: () => { /* log */ });
}
```

**States:**
- **Closed** – Normal operation.
- **Open** – Fails immediately for `durationOfBreak`.
- **Half‑Open** – Allows one trial request; if successful, closes circuit.

**Advanced – combine with retry and timeout:**
```csharp
services.AddHttpClient("api")
    .AddPolicyHandler(Policy.TimeoutAsync<HttpResponseMessage>(TimeSpan.FromSeconds(5)))
    .AddPolicyHandler(Policy.Handle<TimeoutRejectedException>()
        .RetryAsync(2))
    .AddPolicyHandler(GetCircuitBreakerPolicy());
```

### 35. How do you ensure idempotency in a distributed microservices architecture?

**Answer:**  
Idempotency ensures that processing the same request multiple times has the same effect as once. Essential for retries and at‑least‑once delivery.

**Implementation strategies:**

1. **Idempotency Key (most common)**
   - Client generates a unique key (e.g., `Guid`) and sends in header `Idempotency-Key`.
   - Server stores the key with the result in a cache/database.
   - If key seen before, return cached response without re‑processing.

```csharp
[HttpPost("orders")]
public async Task<IActionResult> CreateOrder([FromBody] OrderRequest request, 
    [FromHeader(Name = "Idempotency-Key")] string idempotencyKey)
{
    var existing = await _cache.GetAsync<OrderResponse>(idempotencyKey);
    if (existing != null) return Ok(existing);
    
    var order = await _orderService.CreateAsync(request);
    await _cache.SetAsync(idempotencyKey, order, TimeSpan.FromHours(24));
    return Ok(order);
}
```

2. **Database‑enforced uniqueness**
   - Use unique constraints on business keys (e.g., `OrderNumber`) to prevent duplicates.
   - Handle `SqlException` with duplicate key violation gracefully.

3. **Outbox pattern** – ensures messages are published exactly once.

### 37. How do you diagnose and resolve thread‑pool starvation in ASP.NET Core?

**Answer:**  
Thread‑pool starvation occurs when all threads are blocked on synchronous operations, preventing new work from being scheduled.

**Symptoms:**
- High latency under load.
- `ThreadPool.GetAvailableThreads(out worker, out completion)` shows low available threads.
- Logs show `"Heartbeat took longer than 00:00:01"` warnings.

**Causes:**
- Blocking calls like `.Result`, `.Wait()`, `Task.Run` with sync code.
- Long‑running synchronous CPU work.

**Solutions:**
1. **Use async all the way** – avoid sync‑over‑async.
2. **Set minimum thread pool threads** as a temporary mitigation:
```csharp
ThreadPool.SetMinThreads(200, 200);
```
3. **Identify blocking calls with tools like `dotnet-counters` or PerfView.**

**Example of sync‑over‑async anti‑pattern:**
```csharp
// BAD: Blocks thread pool thread
var result = SomeAsyncMethod().Result;
```

**Fix:**
```csharp
var result = await SomeAsyncMethod();
```

### 38. Explain `ValueTask` and when to use it instead of `Task`.

**Answer:**  
`ValueTask` is a discriminated union that can wrap a synchronous result or an asynchronous `Task`. It reduces allocations when the operation completes synchronously most of the time.

**Use cases:**
- High‑frequency methods (e.g., caching layer).
- Methods that often return immediately (e.g., reading from `MemoryCache`).

**Example:**
```csharp
public ValueTask<Product> GetProductAsync(int id)
{
    if (_cache.TryGetValue(id, out Product product))
        return new ValueTask<Product>(product);
    
    return new ValueTask<Product>(FetchFromDatabaseAsync(id));
}
```

**Caveats:**
- A `ValueTask` should **never** be awaited multiple times; it may be consumed only once.
- Cannot be cached or stored as a field.
- Use `ValueTask<T>` only when profiling proves allocation savings are significant.

```csharp
    {
        entry.SlidingExpiration = TimeSpan.FromMinutes(5);
        return await _dbContext.Products.FindAsync(id);
    });
}
```

**2. Distributed caching (`IDistributedCache` with Redis):**
```csharp
public async Task<string> GetCachedValueAsync(string key)
{
    var bytes = await _distributedCache.GetAsync(key);
    if (bytes != null) return Encoding.UTF8.GetString(bytes);
    
    var value = await FetchFromSourceAsync();
    await _distributedCache.SetAsync(key, Encoding.UTF8.GetBytes(value),
        new DistributedCacheEntryOptions { AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(10) });
    return value;
}
```

**3. Response caching middleware:**
```csharp
app.UseResponseCaching();
[HttpGet]
[ResponseCache(Duration = 60, Location = ResponseCacheLocation.Any)]
public IActionResult Get() => Ok(DateTime.Now);
```

**4. Output caching (`.NET 7+`):**
```csharp
builder.Services.AddOutputCache();
app.UseOutputCache();

[HttpGet("products")]
[OutputCache(Duration = 30)]
public async Task<List<Product>> GetProducts() => await _dbContext.Products.ToListAsync();
```

**Cache invalidation:** Use events, TTL, or explicit `Remove()`.

### 40. What are the key considerations for designing a high‑availability system with .NET?

**Answer:**  

1. **Stateless services** – store state externally (Redis, SQL, blob storage).
2. **Resilience patterns** – retry, circuit breaker, bulkhead isolation, timeout.
3. **Graceful degradation** – fallback responses when dependencies fail.
4. **Health checks** – enable automatic restarts and traffic management.
5. **Horizontal scaling** – design for multiple instances (no sticky sessions unless necessary).
6. **Database read replicas** – separate read and write workloads.
7. **Idempotent operations** – safe to retry.
8. **Monitoring and observability** – structured logging, metrics, distributed tracing with OpenTelemetry.

**Example – Polly policies for resilience:**
```csharp
services.AddHttpClient("external-api")
    .AddPolicyHandler(Policy<HttpResponseMessage>
        .Handle<HttpRequestException>()
        .OrResult(r => !r.IsSuccessStatusCode)
        .WaitAndRetryAsync(3, retryAttempt => TimeSpan.FromSeconds(Math.Pow(2, retryAttempt))))
    .AddPolicyHandler(Policy.TimeoutAsync<HttpResponseMessage>(TimeSpan.FromSeconds(3)));
```
