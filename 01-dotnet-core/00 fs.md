## 1. How does the .NET Garbage Collector work, and what are the implications for high‑performance applications?

**Answer:**

The .NET Garbage Collector (GC) is a generational, tracing collector. It divides the managed heap into three generations:

- **Gen 0** – short‑lived objects (e.g., local variables inside methods)
- **Gen 1** – buffer between short‑lived and long‑lived objects
- **Gen 2** – long‑lived objects (e.g., static data, singletons)

**Collection Process:**
1. When Gen 0 fills up, a **Gen 0 collection** occurs. Surviving objects are promoted to Gen 1.
2. If Gen 1 becomes full, a **Gen 1 collection** runs; survivors move to Gen 2.
3. A **Gen 2 collection** (full GC) is expensive and usually includes **compaction** of the Large Object Heap (LOH) unless LOH compaction is explicitly disabled.

**Modes:**
- **Workstation GC** – optimized for interactive applications (e.g., desktop, ASP.NET with low latency).
- **Server GC** – uses a separate heap and GC thread per CPU core, ideal for high‑throughput server apps (default for ASP.NET Core).

**Implications for high‑performance apps:**
- **Avoid Gen 2 collections** by minimizing long‑lived object allocations and avoiding large object pinning.
- **Use `ArrayPool<T>` and `MemoryPool<T>`** to reuse buffers and reduce allocations.
- **Prefer `Span<T>` and `stackalloc`** for temporary data that doesn’t need heap allocation.
- **Monitor GC metrics** with `GC.GetGCMemoryInfo()` or tools like PerfView / dotnet‑trace.

**Example: Avoiding LOH allocations**
```csharp
// BAD: byte array > 85,000 bytes goes to LOH
byte[] largeBuffer = new byte[100_000];

// GOOD: Use ArrayPool to rent and return
byte[] rented = ArrayPool<byte>.Shared.Rent(100_000);
try { /* use rented */ }
finally { ArrayPool<byte>.Shared.Return(rented); }
```

## 2. Explain the difference between `IQueryable` and `IEnumerable` in Entity Framework Core. Why does it matter?

**Answer:**

| Aspect                 | `IEnumerable<T>`                                   | `IQueryable<T>`                                          |
|------------------------|----------------------------------------------------|----------------------------------------------------------|
| **Namespace**          | `System.Collections.Generic`                       | `System.Linq`                                            |
| **Execution**          | In‑memory (LINQ‑to‑Objects)                        | Deferred / translated to SQL (LINQ‑to‑Entities)          |
| **Where filter runs**  | After data is fetched (client‑side)                | On the database server (SQL `WHERE` clause)              |
| **Performance**        | Potentially large data transfer and memory usage   | Efficient – only necessary rows/columns returned         |

**Practical Impact:**
```csharp
// IEnumerable: All Products are loaded into memory, then filtered
IEnumerable<Product> products = dbContext.Products;
var expensive = products.Where(p => p.Price > 1000).ToList();

// IQueryable: SQL WHERE Price > 1000 is executed on the server
IQueryable<Product> query = dbContext.Products;
var expensive = query.Where(p => p.Price > 1000).ToList();
```

**Advanced Pitfall:** Methods that force immediate execution (`ToList()`, `Count()`, `FirstOrDefault()`) should be called **after** all filtering and projection have been applied via `IQueryable`. Using `AsEnumerable()` in the middle switches to client‑side evaluation, which can cause unintended full table scans.

## 3. Describe the middleware pipeline in ASP.NET Core and how you would implement a custom middleware for request/response logging.

**Answer:**

The ASP.NET Core request pipeline is a chain of middleware components. Each middleware:
- Receives an `HttpContext`
- Performs some work
- Calls the next middleware via `await _next(context)`
- Optionally does work after the next middleware returns.

**Built‑in examples:** UseStaticFiles, UseRouting, UseAuthentication, UseAuthorization, UseEndpoints.

**Custom Logging Middleware:**
```csharp
public class RequestResponseLoggingMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<RequestResponseLoggingMiddleware> _logger;

    public RequestResponseLoggingMiddleware(RequestDelegate next, ILogger<RequestResponseLoggingMiddleware> logger)
    {
        _next = next;
        _logger = logger;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        // Log Request
        context.Request.EnableBuffering(); // Allows reading body multiple times
        var requestBody = await new StreamReader(context.Request.Body).ReadToEndAsync();
        context.Request.Body.Position = 0;
        _logger.LogInformation("Request {Method} {Path} {Body}", 
            context.Request.Method, context.Request.Path, requestBody);

        // Capture response
        var originalBodyStream = context.Response.Body;
        using var responseBody = new MemoryStream();
        context.Response.Body = responseBody;

        await _next(context);

        // Log Response
        context.Response.Body.Seek(0, SeekOrigin.Begin);
        var responseText = await new StreamReader(context.Response.Body).ReadToEndAsync();
        context.Response.Body.Seek(0, SeekOrigin.Begin);
        _logger.LogInformation("Response {StatusCode} {Body}", 
            context.Response.StatusCode, responseText);

        await responseBody.CopyToAsync(originalBodyStream);
    }
}

// Registration in Program.cs
app.UseMiddleware<RequestResponseLoggingMiddleware>();
```

**Important:** The order of middleware registration matters. Logging middleware should be placed early to capture everything, but after static file middleware if you don’t want to log those requests.

## 4. What are `Span<T>` and `Memory<T>`? Provide a scenario where they significantly improve performance.

**Answer:**

- **`Span<T>`** – A stack‑only, ref struct that provides a type‑safe view over contiguous memory (managed array, unmanaged memory, stackalloc). Cannot be stored on the heap (hence no fields in classes, no async methods).
- **`Memory<T>`** – A heap‑allocatable counterpart that can represent the same memory segments and is safe to use in async methods and class fields.

**Performance Benefits:**
- Avoid allocations and copying when slicing arrays or strings.
- Work directly with unmanaged memory or stack‑allocated buffers.
- Reduce GC pressure.

**Scenario: Parsing a large comma‑separated string without `Substring` allocations.**
```csharp
public static int SumCsvValues(ReadOnlySpan<char> csvLine)
{
    int sum = 0;
    foreach (var segment in csvLine.Split(','))
    {
        // No allocation; segment is a ReadOnlySpan<char>
        if (int.TryParse(segment, out int value))
            sum += value;
    }
    return sum;
}

// Usage:
string bigCsv = "100,200,300,400,500";
int total = SumCsvValues(bigCsv.AsSpan());
```
Without `Span`, typical code would call `string.Split` creating an array of substrings and allocating many small strings.

## 5. How would you design a distributed caching layer for a .NET microservices ecosystem?

**Answer:**

**Goals:** Low latency, high availability, consistency across services.

**Common Stack:**
- **Cache Provider:** Redis (Azure Cache for Redis, AWS ElastiCache)
- **.NET Library:** `Microsoft.Extensions.Caching.StackExchangeRedis`

**Design Considerations:**

1. **Cache Invalidation Strategy**
   - **TTL (Time‑To‑Live)** – Use appropriate expiration based on data volatility.
   - **Cache‑Aside Pattern** – Application checks cache first; on miss, loads from DB and populates cache.
   - **Write‑Through / Write‑Behind** – Update cache synchronously or asynchronously when data changes.
   - **Event‑Driven Invalidation** – Services publish `EntityUpdated` events; interested services evict their cached keys.

2. **Key Naming Convention**
   - Use hierarchical keys: `{service}:{entity}:{id}` e.g., `catalog:product:123`.

3. **Serialization**
   - Use binary formats (e.g., Protobuf, MessagePack) to reduce payload size and serialization overhead.

4. **Resilience**
   - Implement **circuit breaker** and **fallback** policies (Polly) to handle Redis outages gracefully.

5. **Scaling**
   - Redis Cluster or Sentinel for high availability.
   - Use `IDistributedCache` abstraction to swap providers if needed.

**Sample Registration:**
```csharp
services.AddStackExchangeRedisCache(options =>
{
    options.Configuration = "redis-connection-string";
    options.InstanceName = "MyApp";
});
```

## 6. Compare JWT and OAuth 2.0 with OpenID Connect. How do you securely implement authentication in a SPA + .NET Core API?

**Answer:**

- **JWT (JSON Web Token)** – A token format (RFC 7519). It contains claims and can be signed (JWS) or encrypted (JWE). It is **stateless** – the resource server validates the token without a central authority per request.
- **OAuth 2.0** – An **authorization framework** (RFC 6749). It defines how a client obtains an access token (often a JWT) to access protected resources.
- **OpenID Connect (OIDC)** – An **authentication layer** on top of OAuth 2.0. It adds an ID Token (JWT) containing user identity claims and a UserInfo endpoint.

**Secure SPA + .NET Core API Implementation:**

1. **Use Authorization Code Flow with PKCE** (Proof Key for Code Exchange) – NOT Implicit Flow.  
   - The SPA redirects to the identity provider (e.g., Azure AD B2C, Auth0, IdentityServer).
   - After login, the SPA receives an **authorization code** via redirect URI.
   - SPA exchanges code + code verifier for **access token** and **refresh token** (backend‑for‑frontend pattern recommended).

2. **Backend‑for‑Frontend (BFF)**  
   - Host the SPA and a lightweight backend proxy together.
   - Tokens are stored in secure, HTTP‑only cookies (not accessible from JavaScript) to mitigate XSS.
   - The BFF handles token exchange and refresh.

3. **.NET Core API Configuration:**
```csharp
services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.Authority = "https://your-identity-server";
        options.Audience = "api1";
        options.TokenValidationParameters = new TokenValidationParameters
        {
            ValidateIssuer = true,
            ValidateAudience = true,
            ValidateLifetime = true,
            ClockSkew = TimeSpan.Zero // Reduce default 5 min skew
        };
    });
```

4. **Token Lifetime:**
   - Access token: short‑lived (5‑15 min)
   - Refresh token: longer‑lived (days) with rotation enabled.

## 7. What is the purpose of `IHostedService` and `BackgroundService`? How do you handle graceful shutdown in a background task?

**Answer:**

`IHostedService` defines a contract for long‑running background tasks in ASP.NET Core. `BackgroundService` is a base class that implements `IHostedService` and provides a simpler `ExecuteAsync` method.

**Implementation:**
```csharp
public class TimedHostedService : BackgroundService
{
    private readonly ILogger<TimedHostedService> _logger;
    private readonly PeriodicTimer _timer = new(TimeSpan.FromSeconds(30));

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        try
        {
            while (await _timer.WaitForNextTickAsync(stoppingToken))
            {
                // Do work
                _logger.LogInformation("Worker running at: {time}", DateTimeOffset.Now);
            }
        }
        catch (OperationCanceledException)
        {
            _logger.LogInformation("Service is stopping.");
        }
    }
}
```

**Graceful Shutdown:**
- ASP.NET Core signals application shutdown via a `CancellationToken` passed to `StartAsync` (for `IHostedService`) or `ExecuteAsync` (for `BackgroundService`).
- The token is triggered when the host receives `SIGTERM` (e.g., `CTRL+C`, Docker stop).
- The host waits for `ShutdownTimeout` (default 30 seconds) before aborting. Ensure your background task respects the token.

**Best Practices:**
- Check `stoppingToken.IsCancellationRequested` frequently.
- Use async methods that accept cancellation tokens.
- Avoid blocking calls; use `Task.Delay(..., cancellationToken)` instead of `Thread.Sleep`.

## 8. Explain how to implement the Outbox Pattern with EF Core and a message broker (e.g., RabbitMQ or Azure Service Bus).

**Answer:**

**Problem:** In a distributed transaction, you update the database and publish an event. If the publish fails after the database commit, the system becomes inconsistent.

**Outbox Pattern Solution:**
1. Instead of publishing the event immediately, write the event to an **Outbox table** within the same database transaction.
2. A separate **publisher process** reads the outbox table and sends messages to the broker.
3. After successful publish, mark the outbox record as processed (or delete it).

**Implementation Steps:**

**1. Outbox Entity:**
```csharp
public class OutboxMessage
{
    public Guid Id { get; set; }
    public string EventType { get; set; }
    public string EventData { get; set; } // JSON
    public DateTime OccurredOn { get; set; }
    public DateTime? ProcessedOn { get; set; }
}
```

**2. Save in Transaction:**
```csharp
public async Task PlaceOrderAsync(Order order)
{
    using var transaction = await _dbContext.Database.BeginTransactionAsync();
    _dbContext.Orders.Add(order);
    
    var outboxMessage = new OutboxMessage
    {
        Id = Guid.NewGuid(),
        EventType = nameof(OrderPlacedEvent),
        EventData = JsonSerializer.Serialize(new OrderPlacedEvent(order.Id)),
        OccurredOn = DateTime.UtcNow
    };
    _dbContext.OutboxMessages.Add(outboxMessage);
    
    await _dbContext.SaveChangesAsync();
    await transaction.CommitAsync();
}
```

**3. Background Publisher (HostedService):**
```csharp
protected override async Task ExecuteAsync(CancellationToken stoppingToken)
{
    while (!stoppingToken.IsCancellationRequested)
    {
        var messages = await _dbContext.OutboxMessages
            .Where(m => m.ProcessedOn == null)
            .Take(100)
            .ToListAsync(stoppingToken);
            
        foreach (var msg in messages)
        {
            // Publish to broker (RabbitMQ, Service Bus)
            await _messageBus.PublishAsync(msg.EventType, msg.EventData);
            msg.ProcessedOn = DateTime.UtcNow;
        }
        
        await _dbContext.SaveChangesAsync(stoppingToken);
        await Task.Delay(2000, stoppingToken);
    }
}
```

**Alternative:** Use **Transactional Outbox** with **Change Data Capture (CDC)** or **Debezium** for more robust at‑least‑once delivery.

## 9. How does Angular's change detection differ from React's reconciliation? What are the performance implications in large enterprise applications?

**Answer:**

| Aspect                     | Angular (Zone.js based)                                    | React (Virtual DOM)                                        |
|----------------------------|------------------------------------------------------------|------------------------------------------------------------|
| **Mechanism**              | Monkey‑patches browser APIs (setTimeout, events) via Zone.js. After any async operation, it runs change detection on the component tree. | Builds an in‑memory Virtual DOM representation. On state change, creates a new VDOM tree and diffs with previous tree, then updates the real DOM only where necessary. |
| **Granularity**            | Checks every component from root down (unless `OnPush` strategy is used). | Component‑level re‑rendering, but diffing reduces DOM updates. |
| **Optimization Default**   | `ChangeDetectionStrategy.Default` – checks every component on every event. | React components re‑render when state/props change; `React.memo` / `PureComponent` skip rendering if props shallow‑equal. |
| **Performance in Large Apps** | Can become slow if many components use `Default` strategy and events fire frequently. Requires explicit use of `OnPush` and `ChangeDetectorRef` for manual control. | React's diffing is efficient, but deep component trees can still cause unnecessary VDOM creations unless `memo` is used. |

**Angular Performance Best Practices:**
- Use `ChangeDetectionStrategy.OnPush` for all smart/presentational components.
- Immutable data patterns or `Observable` with `async` pipe to trigger change detection only when new references arrive.
- Detach change detector for heavy components and manually call `detectChanges()`.

**React Performance Best Practices:**
- Use `React.memo` for pure functional components.
- Use `useMemo` and `useCallback` to prevent unnecessary re‑computations and function recreations.
- Virtualize long lists with `react‑window`.

**Verdict:** Both frameworks can handle enterprise apps with proper optimization. Angular requires more deliberate change detection configuration; React's default re‑rendering can be wasteful but is often mitigated by the VDOM diffing efficiency.

## 10. Describe how you would set up a CI/CD pipeline for a containerized .NET Core microservices app using GitHub Actions and Azure Kubernetes Service.

**Answer:**

**High‑Level Flow:**
1. **Build & Test** – Compile .NET services, run unit/integration tests.
2. **Containerize** – Build Docker images for each microservice.
3. **Push to Registry** – Push images to Azure Container Registry (ACR).
4. **Deploy to AKS** – Update Kubernetes manifests and apply.

**GitHub Actions Workflow (.github/workflows/deploy.yml):**
```yaml
name: Build and Deploy to AKS

on:
  push:
    branches: [ main ]

env:
  ACR_NAME: myacr
  AKS_CLUSTER: myakscluster
  AKS_RESOURCE_GROUP: myrg

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v3
    
    - name: Setup .NET
      uses: actions/setup-dotnet@v3
      with:
        dotnet-version: '8.0.x'
    
    - name: Restore dependencies
      run: dotnet restore
    
    - name: Build
      run: dotnet build --configuration Release --no-restore
    
    - name: Test
      run: dotnet test --no-build --configuration Release
    
    - name: Azure Login
      uses: azure/login@v1
      with:
        creds: ${{ secrets.AZURE_CREDENTIALS }}
    
    - name: Docker Build and Push
      uses: azure/docker-login@v1
      with:
        login-server: ${{ env.ACR_NAME }}.azurecr.io
        username: ${{ secrets.ACR_USERNAME }}
        password: ${{ secrets.ACR_PASSWORD }}
    
    - run: |
        docker build -t $ACR_NAME.azurecr.io/catalog-api:${{ github.sha }} -f src/Catalog.API/Dockerfile .
        docker push $ACR_NAME.azurecr.io/catalog-api:${{ github.sha }}
    
    - name: Set AKS context
      uses: azure/aks-set-context@v3
      with:
        resource-group: ${{ env.AKS_RESOURCE_GROUP }}
        cluster-name: ${{ env.AKS_CLUSTER }}
    
    - name: Deploy to AKS
      uses: azure/k8s-deploy@v4
      with:
        manifests: |
          k8s/catalog-api-deployment.yaml
          k8s/catalog-api-service.yaml
        images: |
          ${{ env.ACR_NAME }}.azurecr.io/catalog-api:${{ github.sha }}
```

**Advanced Enhancements:**
- Use **Helm** for templating Kubernetes manifests.
- Implement **Blue/Green** or **Canary** deployments via **Flagger** or **Argo Rollouts**.
- Integrate **Azure Key Vault** for secrets.
- Add **Infrastructure as Code** (Terraform/Bicep) for cluster provisioning.

## 11. What are the trade‑offs between using gRPC and RESTful APIs in a .NET full‑stack application?

**Answer:**

| Feature                | gRPC                                                    | REST/JSON over HTTP                                     |
|------------------------|---------------------------------------------------------|---------------------------------------------------------|
| **Protocol**           | HTTP/2 (binary framing, multiplexing, server push)      | HTTP/1.1 (text‑based) or HTTP/2                         |
| **Payload**            | Protocol Buffers (binary, small, strongly typed)        | JSON/XML (text, larger, less strict)                    |
| **Performance**        | **Higher throughput, lower latency** due to binary and HTTP/2 multiplexing | Good enough for most apps; may suffer from head‑of‑line blocking on HTTP/1.1 |
| **Browser Support**    | Limited; requires gRPC‑Web for browsers                 | Universal                                               |
| **Tooling / Debugging**| Requires protoc compiler; less human‑readable wire format | Easy to inspect with browser dev tools, curl           |
| **Streaming**          | First‑class support for client/server/bi‑directional streaming | Requires WebSockets or Server‑Sent Events              |
| **Code Generation**    | Strongly‑typed client and server stubs from .proto       | OpenAPI/Swagger provides documentation, but client generation is optional |

**When to choose gRPC:**
- Internal microservice communication where low latency and high throughput matter.
- Real‑time bidirectional streaming (e.g., chat, live data feeds).
- Polyglot environments (gRPC supports many languages).

**When to choose REST:**
- Public APIs consumed by third‑party developers.
- Browsers directly calling the API (unless using gRPC‑Web with additional complexity).
- Simpler development and debugging needs.

**Hybrid Approach:** Expose both gRPC and REST endpoints from the same ASP.NET Core service using `Microsoft.AspNetCore.Grpc.JsonTranscoding` (experimental) or separate endpoints.

## 12. How do you manage database migrations across multiple environments and teams with EF Core?

**Answer:**

**Challenges:** Multiple branches with schema changes, merging conflicts, deployment to dev/staging/prod.

**Best Practices:**

1. **Use EF Core Migrations with Idempotent SQL Scripts:**
   - Generate idempotent scripts via `dotnet ef migrations script --idempotent`. These scripts contain `IF NOT EXISTS` checks for each migration.
   - Apply scripts during CI/CD with `dotnet ef database update` or execute the script manually in production.

2. **Separate Migration Generation from Application Startup:**
   - Do NOT run `context.Database.Migrate()` on production app startup (can cause issues in multi‑instance deployments).
   - Instead, use a separate **migration job** in your CI/CD pipeline.

3. **Branching Strategy:**
   - Each feature branch may create a migration. Merge to `main` results in multiple migrations. After merging, generate a **merge migration** using:
     ```bash
     dotnet ef migrations add MergeMigrationFromFeatures
     ```
   - This squashes the pending model changes into a single migration, avoiding timestamp ordering issues.

4. **Database per Environment:**
   - Use environment‑specific connection strings.
   - Apply migrations to dev automatically on each commit (fast feedback).
   - For staging/prod, run migrations as a controlled step before new app version goes live.

5. **Handling Breaking Schema Changes:**
   - Use **expand‑contract pattern** (a.k.a. blue‑green database changes):
     - Add new column (nullable) → deploy app that writes to both old and new.
     - Backfill data → deploy app that reads from new column.
     - Drop old column → final deployment.

6. **Use a Migration Bundle (EF Core 6+):**
   - `dotnet ef migrations bundle` creates a self‑contained executable that applies migrations. This is ideal for CI/CD because it avoids installing the .NET SDK on the deployment agent.

## 13. Explain the principles of the Clean Architecture template in .NET and how it facilitates testability.

**Answer:**

Clean Architecture (popularized by Jason Taylor’s template) organizes code into concentric circles with **dependencies pointing inward**.

**Layers (inner to outer):**
1. **Domain** – Entities, value objects, domain events, interfaces for repositories (no external dependencies).
2. **Application** – Use cases (commands/queries with MediatR), DTOs, business logic orchestration. Depends only on Domain.
3. **Infrastructure** – Implementation of repository interfaces, database context, external services (email, file storage). Depends on Application and Domain.
4. **WebUI** – Controllers, Minimal APIs, front‑end assets. Depends on Application and Infrastructure (for DI registration).

**Key Principles:**
- **Dependency Inversion:** High‑level modules (Application) do not depend on low‑level (Infrastructure). Instead, both depend on abstractions defined in inner layers.
- **Separation of Concerns:** Each layer has a clear responsibility.
- **Testability:** Because business logic lives in Application and Domain (without external concerns), unit tests can run without a database or web server.

**Facilitates Testability:**
- **Unit Tests** can test Command/Query Handlers by mocking repository interfaces (`IMock<IRepository>`).
- **Integration Tests** can use a real test database (e.g., SQLite in‑memory) by substituting Infrastructure implementations via `CustomWebApplicationFactory`.
- **End‑to‑End Tests** can run against a fully deployed instance.

**Sample Use Case Handler Test:**
```csharp
[Test]
public async Task CreateOrderCommand_ValidInput_CreatesOrder()
{
    var mockRepo = new Mock<IOrderRepository>();
    var handler = new CreateOrderCommandHandler(mockRepo.Object);
    var command = new CreateOrderCommand { CustomerId = 1, Total = 100 };
    
    var result = await handler.Handle(command, CancellationToken.None);
    
    Assert.IsTrue(result.Success);
    mockRepo.Verify(r => r.AddAsync(It.IsAny<Order>()), Times.Once);
}
```

## 14. What are the best practices for handling large file uploads in ASP.NET Core without exhausting server resources?

**Answer:**

**Problems with default file upload:**
- Entire file is buffered in memory or on disk before processing.
- Large files can cause `OutOfMemoryException` or fill up disk space.
- Long‑running requests block threads.

**Best Practices:**

1. **Disable Form Value Model Binding for Files:** Use `Microsoft.AspNetCore.WebUtilities.MultipartReader` to read the stream sequentially.
2. **Stream the File Directly to Storage:** Write chunks to Azure Blob Storage, AWS S3, or disk without holding the entire file in memory.
3. **Use `IFormFile` with `EnableBuffering` False:** By default, ASP.NET Core buffers files >64KB. Override with `[DisableFormValueModelBinding]` attribute.
4. **Implement Request Size Limits:**
   ```csharp
   [RequestSizeLimit(100_000_000)] // 100 MB
   [RequestFormLimits(MultipartBodyLengthLimit = 100_000_000)]
   public async Task<IActionResult> UploadLargeFile()
   ```
5. **Use Streaming Endpoint with `FileStreamResult` for Download:**
   ```csharp
   public IActionResult Download()
   {
       var stream = new FileStream("largefile.zip", FileMode.Open);
       return File(stream, "application/octet-stream", "largefile.zip");
   }
   ```
6. **Offload to Background Service:** After receiving the file, queue a message for processing (e.g., virus scan, thumbnail generation) using a durable queue like Azure Storage Queue.

**Example: Streaming Upload with MultipartReader**
```csharp
[HttpPost]
[DisableFormValueModelBinding]
public async Task<IActionResult> Upload()
{
    var boundary = Request.GetMultipartBoundary();
    var reader = new MultipartReader(boundary, Request.Body);
    var section = await reader.ReadNextSectionAsync();
    
    while (section != null)
    {
        var fileSection = section.AsFileSection();
        if (fileSection != null)
        {
            var filePath = Path.GetTempFileName();
            using var stream = new FileStream(filePath, FileMode.Create);
            await fileSection.FileStream.CopyToAsync(stream);
            // Or better: await blobClient.UploadAsync(fileSection.FileStream);
        }
        section = await reader.ReadNextSectionAsync();
    }
    return Ok();
}
```

## 15. How would you implement real‑time notifications in a .NET + Angular/React application using SignalR and Azure SignalR Service?

**Answer:**

**Architecture Overview:**
- **Client (Angular/React)** connects to **Azure SignalR Service** via WebSocket.
- **.NET Core API** acts as a SignalR Hub and sends messages to connected clients.
- **Azure SignalR Service** handles connection management, scaling, and persistence (removing the need for sticky sessions).

**Implementation Steps:**

**1. Add SignalR to ASP.NET Core:**
```csharp
services.AddSignalR().AddAzureSignalR(Configuration["Azure:SignalR:ConnectionString"]);
app.MapHub<NotificationHub>("/notificationHub");
```

**2. Define Hub:**
```csharp
public class NotificationHub : Hub
{
    public async Task SendNotification(string user, string message)
    {
        await Clients.User(user).SendAsync("ReceiveNotification", message);
    }
}
```

**3. Authenticate Users (JWT):**
```csharp
services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options => { /* ... */ });

// In Hub, access user via Context.User
public override async Task OnConnectedAsync()
{
    var userId = Context.UserIdentifier; // maps to ClaimTypes.NameIdentifier by default
    await Groups.AddToGroupAsync(Context.ConnectionId, userId);
    await base.OnConnectedAsync();
}
```

**4. Angular Client Setup:**
```typescript
import * as signalR from '@microsoft/signalr';

const connection = new signalR.HubConnectionBuilder()
  .withUrl('https://your-api/notificationHub', {
    accessTokenFactory: () => localStorage.getItem('access_token')
  })
  .withAutomaticReconnect()
  .build();

connection.on('ReceiveNotification', (message: string) => {
  console.log('Notification:', message);
});

connection.start().catch(err => console.error(err));
```

**5. Sending Notifications from Server (e.g., after a database event):**
```csharp
public class OrderService
{
    private readonly IHubContext<NotificationHub> _hubContext;
    public OrderService(IHubContext<NotificationHub> hubContext) => _hubContext = hubContext;

    public async Task PlaceOrderAsync(Order order)
    {
        // Save order...
        await _hubContext.Clients.User(order.UserId)
            .SendAsync("ReceiveNotification", $"Order {order.Id} placed.");
    }
}
```

**Azure SignalR Service Benefits:**
- **Auto‑scaling:** Handles millions of concurrent connections without managing your own WebSocket servers.
- **Resilience:** Built‑in reconnection and message buffering.
- **Global Reach:** Deploy in multiple regions for low‑latency.
